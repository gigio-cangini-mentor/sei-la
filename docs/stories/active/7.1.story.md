## Story

**As a** JARVIS Cloud Brain operator,
**I want** rate limiting on all Cloud Brain endpoints,
**so that** the system is protected against abuse, accidental loops, and API cost overruns.

## Acceptance Criteria

1. All `/api/brain/*` endpoints enforce per-IP rate limits (configurable, default: 30 req/min for execute, 60 req/min for read-only)
2. `/api/brain/execute` has an additional per-session rate limit (default: 10 req/min) to prevent runaway agent loops
3. WebSocket connections enforce a message rate limit (default: 20 msg/min per client)
4. Rate limit headers are returned on every response (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`)
5. When rate limit is exceeded, return `429 Too Many Requests` with a JSON body containing `retryAfter` in seconds
6. Anthropic API calls have a cost guard: max $5/hour per session (configurable via env `BRAIN_COST_LIMIT_HOURLY`)
7. Rate limit config is stored in environment variables with sensible defaults (no hard-coded values)
8. All rate limiting works correctly behind Traefik reverse proxy (uses `X-Forwarded-For` for real IP)

## Dev Notes

### Architecture Context

[Source: apps/jarvis-hub/server.js — Express + WS server]

Server.js e o ponto de entrada. Todas as rotas `/api/brain/*` e `/api/jarvis/*` passam pelo Express. WebSocket connections sao gerenciadas pelo `ws` package. O middleware de rate limiting deve ser aplicado antes de qualquer handler.

### Implementacao Recomendada

Sliding window counter em memoria (Map de IP → timestamps). Sem dependencia externa (Redis). Para producao single-instance (VPS), in-memory e suficiente.

Para custo da API Anthropic, usar os campos `usage.input_tokens` e `usage.output_tokens` retornados pela API Claude.
