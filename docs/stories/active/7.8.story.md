## Story

**As a** JARVIS Cloud Brain operator,
**I want** health monitoring with alerts for critical failures,
**so that** I know immediately when something goes wrong on the VPS.

## Acceptance Criteria

1. A health endpoint `GET /health` returns system status: server uptime, memory usage, disk usage, Anthropic API status, WebSocket clients count
2. A detailed health endpoint `GET /api/brain/health` returns: last successful API call, agent execution stats, error rate (last 1h), active conversations
3. Server monitors disk space: warn at 80% usage, critical at 90%
4. Server monitors memory: warn at 80% of container limit, critical at 90%
5. Health check failures are broadcast to WebSocket clients as `health_alert` events
6. Cockpit shows a health indicator (green/yellow/red) in the topbar based on health status
7. Health data is collected every 60 seconds and stored for 24h (for trend display)
8. If Anthropic API has been unreachable for > 5 minutes, cockpit shows a banner warning

## Tasks / Subtasks

- [ ] Task 1: Health endpoint — basic (AC: 1)
  - [ ] `GET /health` — lightweight, no auth required (for Docker/Traefik health checks)
  - [ ] Return: `{ status: "ok"|"degraded"|"down", uptime, memory_mb, disk_percent, ws_clients }`
  - [ ] Response time target: < 50ms
- [ ] Task 2: Health endpoint — detailed (AC: 2)
  - [ ] `GET /api/brain/health` — auth required
  - [ ] Return: last API call timestamp, agent stats (invocations/errors per agent), error rate, active conversations count
  - [ ] Include per-agent stats from last 1h
- [ ] Task 3: Resource monitoring (AC: 3, 4)
  - [ ] Monitor disk usage via `os` module or `df` command
  - [ ] Monitor memory via `process.memoryUsage()`
  - [ ] Thresholds: warn at 80%, critical at 90%
  - [ ] Check every 60 seconds
- [ ] Task 4: Alert broadcasting (AC: 5, 8)
  - [ ] On threshold breach: broadcast `health_alert` event via WebSocket
  - [ ] Event shape: `{ type: "health_alert", level: "warn"|"critical", component, message }`
  - [ ] Track Anthropic API connectivity (timestamp of last successful call)
  - [ ] If > 5 min since last success: broadcast API_UNREACHABLE alert
- [ ] Task 5: Cockpit health indicator (AC: 6, 8)
  - [ ] Add health dot to topbar (green/yellow/red)
  - [ ] Poll `/health` every 30s (or use WS events)
  - [ ] Yellow: any warn-level alert
  - [ ] Red: any critical-level alert or API unreachable
  - [ ] Click on indicator shows health details popup
  - [ ] API unreachable banner: full-width yellow bar below topbar
- [ ] Task 6: Health history (AC: 7)
  - [ ] Store health snapshots in memory (circular buffer, 1440 entries = 24h at 60s interval)
  - [ ] `GET /api/brain/health/history` returns last N snapshots
  - [ ] Used by cockpit for sparkline/trend display (optional in this story)

## Dev Notes

### Architecture Context

[Source: apps/jarvis-hub/server.js — Express server on VPS]

Atualmente nao existe health check formal. Docker/Traefik pode usar `GET /health` para determinar se o container esta vivo. O cockpit nao tem indicador de saude.

### Docker Health Check

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1
```
