## Story

**As a** JARVIS Cloud Brain operator,
**I want** a comprehensive security audit of the entire Cloud Brain surface,
**so that** all known vulnerability classes are identified and mitigated before production use.

## Acceptance Criteria

1. Audit covers all OWASP Top 10 categories relevant to the Cloud Brain (injection, broken auth, SSRF, etc.)
2. All tool blocklists are reviewed and updated: BLOCKED_PATTERNS, BLOCKED_PATHS, BLOCKED_COMMANDS per agent
3. Path traversal vectors are tested and blocked: `..`, symlinks, URL encoding (`%2e%2e`), null bytes
4. Injection vectors are tested: command injection via tool args, prompt injection via user input, JSONL injection in audit logs
5. Authentication review: PIN auth strength, session token entropy, WS auth bypass attempts
6. CORS and CSP headers are properly configured for the cockpit
7. All findings are documented in `docs/security/W7-SECURITY-AUDIT.md` with severity (Critical/High/Medium/Low) and remediation status
8. Zero Critical and zero High findings remain open after remediation

## Dev Notes

### Architecture Context

[Source: apps/jarvis-hub/server.js, cloud-brain.js — security surface]

O Cloud Brain expoe: REST API (Express), WebSocket (ws), tool execution (child_process). Authentication e PIN-based (W2.12). SSL e via Traefik (W5.9). Tool sandboxing e expandido na Story 7.2.

### Superficie de Ataque

1. **REST API** — qualquer endpoint sem rate limit ou auth
2. **WebSocket** — mensagens malformadas, flood, auth bypass
3. **Tool execution** — command injection via args
4. **File operations** — path traversal, symlink escape
5. **Prompt injection** — usuario injetando instrucoes no prompt do agente

- Testes de regressao para cada fix aplicado
- Snapshot dos blocklists para detectar remocao acidental
