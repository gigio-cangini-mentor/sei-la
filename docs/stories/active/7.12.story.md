## Story

**As a** JARVIS Cloud Brain user,
**I want** faster response times and lower resource usage,
**so that** interactions feel snappy and the VPS handles more concurrent usage.

## Acceptance Criteria

1. Agent YAML definitions are cached in memory after first load (TTL: 5 min), reducing file I/O on every request
2. Agent system prompts are cached after build (TTL: 5 min), avoiding repeated CLAUDE.md/roadmap parsing
3. Project knowledge (CLAUDE.md sections, constitution, roadmap) is cached (TTL: 10 min)
4. `detectAgent()` scoring results are cached per unique command (LRU cache, max 100 entries)
5. Lazy loading: agent YAML files are only parsed when that agent is needed (not all at startup)
6. Connection pooling: if multiple agent calls happen in parallel, reuse Anthropic client instance
7. Response time for `POST /api/brain/execute` improves by at least 20% on repeat commands (measured via before/after benchmark)
8. Memory usage stays under 200MB for the server process under normal operation

## Dev Notes

### Architecture Context

[Source: apps/jarvis-hub/cloud-brain.js — parseAgentYaml(), buildSystemPrompt(), loadProjectKnowledge()]

Cada request lê YAML do disco, parseia, builda system prompt, carrega project knowledge. Em uso normal (1-2 req/min) isso e aceitavel, mas para sessions ativas com multiplos comandos em sequencia, caching reduz significativamente a latencia.

### Cache Implementation

```javascript
class TTLCache {
  constructor(ttlMs) {
    this.cache = new Map();
    this.ttl = ttlMs;
  }
  get(key) {
    const entry = this.cache.get(key);
    if (!entry) return null;
    if (Date.now() - entry.ts > this.ttl) {
      this.cache.delete(key);
      return null;
    }
    return entry.value;
  }
  set(key, value) {
    this.cache.set(key, { value, ts: Date.now() });
  }
  clear() { this.cache.clear(); }
}
```
